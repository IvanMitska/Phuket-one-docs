================================================================================
            ГЛОССАРИЙ ТЕРМИНОВ ДЛЯ ИНВЕСТОРОВ
                    Phuket App
================================================================================

Этот документ объясняет технические термины простым языком для презентации
инвесторам.

================================================================================
1. ОБЩИЕ ТЕРМИНЫ ПРОЕКТА
================================================================================

PRD (Product Requirements Document)
    Документ требований к продукту. Описывает ЧТО мы делаем: какие функции
    будут в приложении, для кого оно предназначено, какие проблемы решает.
    Это "техническое задание" для команды разработки.

MVP (Minimum Viable Product)
    Минимально жизнеспособный продукт. Первая версия приложения с базовым
    набором функций, достаточным для запуска и получения обратной связи
    от реальных пользователей.

Roadmap (Роадмап)
    План разработки по месяцам. Показывает какие функции и когда будут
    готовы. Наш план: 5 месяцев разработки + 3 месяца тестирования.

Sprint (Спринт)
    Двухнедельный цикл работы команды. За каждый спринт выполняется
    конкретный набор задач с измеримым результатом.

================================================================================
2. АРХИТЕКТУРА И ДИЗАЙН СИСТЕМЫ
================================================================================

Архитектура приложения
    "Чертёж" приложения. Описывает как все части системы связаны между
    собой: где хранятся данные, как приложение общается с сервером,
    как устроена безопасность.

System Design (Дизайн системы)
    Высокоуровневое описание как работает вся система целиком.
    Как если бы вы объясняли устройство автомобиля, не вдаваясь
    в детали каждого болта.

Solution Architecture
    Архитектура решения. Описывает какие технологии и сервисы используются
    и как они взаимодействуют. Например: "данные хранятся в облаке Amazon,
    платежи через Stripe, карты через Google Maps".

Software Architecture
    Архитектура программного обеспечения. Более детальное описание
    структуры кода: как организованы файлы, модули, компоненты.

ADR (Architecture Decision Record)
    Запись архитектурного решения. Документ, объясняющий ПОЧЕМУ было
    принято то или иное техническое решение. Помогает новым разработчикам
    понять историю проекта.

RFC (Request for Comments)
    Запрос на комментарии. Предложение по улучшению системы, которое
    обсуждается командой перед реализацией. Способ принятия важных
    технических решений коллективно.

================================================================================
3. ПЛАТФОРМЫ РАЗРАБОТКИ
================================================================================

iOS / SwiftUI
    Разработка для iPhone и iPad. SwiftUI - современный инструмент Apple
    для создания интерфейсов. Позволяет делать красивые, быстрые приложения
    в стиле Apple.

Android / Kotlin
    Разработка для Android-устройств (Samsung, Xiaomi, и др.). Kotlin -
    современный язык программирования от Google, рекомендуемый для Android.

Web / Next.js
    Веб-версия приложения, работающая в браузере. Next.js - популярный
    фреймворк для создания быстрых веб-сайтов. Позволяет пользователям
    работать с сервисом без установки приложения.

Cross-platform (Кроссплатформенность)
    Возможность работы на разных устройствах. Наше приложение будет
    доступно на iOS, Android и в браузере.

================================================================================
4. БЭКЕНД (СЕРВЕРНАЯ ЧАСТЬ)
================================================================================

Backend (Бэкенд)
    Серверная часть приложения. "Мозг" системы, который хранит данные,
    обрабатывает заказы, проверяет платежи. Пользователь его не видит,
    но без него приложение не работает.

API (Application Programming Interface)
    Интерфейс для общения приложения с сервером. Как "язык", на котором
    телефон разговаривает с сервером: "дай мне список отелей",
    "создай бронирование", "проверь оплату".

REST API
    Стандартный способ организации API. Используется в 90% современных
    приложений. Простой, надёжный, хорошо документированный.

База данных (Database)
    Хранилище всей информации: пользователи, отели, бронирования, отзывы.
    Мы используем PostgreSQL - надёжную базу данных, которую используют
    крупнейшие компании мира.

Supabase
    Облачная платформа для бэкенда. Предоставляет базу данных,
    авторизацию пользователей и хранение файлов "из коробки".
    Ускоряет разработку в 3-5 раз.

================================================================================
5. ТЕХНИЧЕСКИЙ ДИЗАЙН
================================================================================

Technical Design Document (TDD)
    Документ технического дизайна. Детальное описание КАК именно
    реализовать функцию: какой код написать, какие данные хранить,
    как обрабатывать ошибки.

UI (User Interface)
    Пользовательский интерфейс. Всё, что видит пользователь: кнопки,
    меню, формы, картинки.

UX (User Experience)
    Пользовательский опыт. Насколько удобно и приятно пользоваться
    приложением. Хороший UX = пользователи возвращаются.

Component (Компонент)
    Переиспользуемый элемент интерфейса. Например, кнопка "Забронировать"
    - это компонент, который используется на многих экранах.

================================================================================
6. ИНТЕГРАЦИИ И СЕРВИСЫ
================================================================================

Stripe
    Международная платёжная система. Принимает карты Visa, Mastercard,
    Apple Pay, Google Pay. Используется Uber, Amazon, Shopify.
    Комиссия ~2.9% + $0.30 за транзакцию.

Google Maps API
    Сервис карт от Google. Показывает карты, строит маршруты,
    ищет адреса. Оплата по количеству запросов.

Push-уведомления
    Сообщения, которые приходят на телефон даже когда приложение закрыто.
    "Ваше бронирование подтверждено", "Водитель приедет через 5 минут".

OAuth / Social Login
    Вход через социальные сети (Google, Apple, Facebook).
    Пользователю не нужно запоминать ещё один пароль.

================================================================================
7. ТЕСТИРОВАНИЕ И КАЧЕСТВО
================================================================================

QA (Quality Assurance)
    Обеспечение качества. Процесс проверки приложения на ошибки
    перед выпуском. 3 месяца нашего плана посвящены QA.

Unit Tests (Юнит-тесты)
    Автоматические тесты отдельных функций. Проверяют, что каждая
    маленькая часть кода работает правильно.

E2E Tests (End-to-End тесты)
    Сквозные тесты. Проверяют весь путь пользователя: от открытия
    приложения до завершения бронирования.

Beta-тестирование
    Тестирование с реальными пользователями перед официальным запуском.
    Позволяет найти проблемы, которые не видны разработчикам.

Security Audit (Аудит безопасности)
    Проверка приложения на уязвимости. Защита данных пользователей
    и платёжной информации - критически важна.

Load Testing (Нагрузочное тестирование)
    Проверка как система работает под нагрузкой. Что будет, если
    1000 человек одновременно откроют приложение?

================================================================================
8. МЕТРИКИ И АНАЛИТИКА
================================================================================

DAU (Daily Active Users)
    Ежедневные активные пользователи. Сколько человек пользуются
    приложением каждый день.

MAU (Monthly Active Users)
    Ежемесячные активные пользователи. Ключевая метрика роста.

Retention (Удержание)
    Процент пользователей, которые возвращаются в приложение.
    Высокий retention = продукт нравится людям.

Conversion (Конверсия)
    Процент пользователей, совершивших целевое действие.
    Например: из 100 посмотревших отель, 5 забронировали = 5% конверсия.

GMV (Gross Merchandise Value)
    Общий объём транзакций через платформу. Если через нас забронировали
    отелей на $100,000 - это наш GMV.

================================================================================
9. ТЕРМИНЫ РАЗРАБОТКИ
================================================================================

Git / GitHub
    Система контроля версий. Хранит всю историю изменений кода.
    Позволяет нескольким разработчикам работать одновременно.

CI/CD (Continuous Integration / Continuous Deployment)
    Автоматическая сборка и публикация приложения. Код проверяется
    и выкладывается автоматически, без ручной работы.

Production (Прод)
    Рабочая версия приложения, которой пользуются реальные клиенты.

Staging
    Тестовая версия, копия продакшена для проверки перед релизом.

Hotfix
    Срочное исправление критической ошибки в продакшене.

================================================================================
10. ОБЛАЧНАЯ ИНФРАСТРУКТУРА
================================================================================

Cloud (Облако)
    Серверы в интернете, которые мы арендуем. Не нужно покупать
    своё оборудование. Платим только за использование.

AWS / Google Cloud / Azure
    Крупнейшие облачные провайдеры. Предоставляют серверы, базы данных,
    хранилища файлов. Надёжность 99.9%+.

CDN (Content Delivery Network)
    Сеть доставки контента. Картинки и файлы хранятся на серверах
    по всему миру - пользователь получает их с ближайшего.

SSL / HTTPS
    Шифрование данных при передаче. Защищает пароли и платёжные
    данные от перехвата. Обязательно для любого приложения.
